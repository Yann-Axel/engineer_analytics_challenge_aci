# Part 2 — Naming conventions

A single set of conventions across the dbt project. Used by every model.

## Layer prefixes

| Prefix | Layer | Purpose |
|---|---|---|
| `stg_` | Staging | 1:1 with sources. Cast types, rename, dedup. No joins. |
| `int_` | Intermediate | Joins and pre-aggregations. May be `ephemeral` if not reused. |
| `dim_` | Dimension | Conformed dimensions exposed by marts. |
| `fct_` | Fact | Event tables, well-defined grain. |
| `ont_` | Ontology | Derived business concepts with declarative rules. |
| `m_` (metrics YAML) | Atomic measure exposed as a simple metric (for ratio composition). |

## Suffixes

| Suffix | Meaning |
|---|---|
| `_sk` | Surrogate key (deterministic, generated via `dbt_utils.generate_surrogate_key`) |
| `_id` | Natural key carried from source for traceability |
| `_date` | Calendar date (no time component) |
| `_at` | Timestamp (date + time) |
| `_usd` | Monetary amount in USD (matches Part 1 unit) |
| `_pct` | Percentage in [0, 1] (e.g. `route_margin_pct`) |
| `_min` | Minutes |
| `_km` | Kilometres |

## Booleans

Always prefixed with `is_` or `has_`. Example: `is_cancelled`, `has_disruption`, `is_strategic`, `is_premium_cabin`, `is_loyalty_member`.

## Casing

- Tables, columns, models: `snake_case`
- YAML keys: `snake_case`
- Surrogate keys are integers (or md5 hex) generated by `dbt_utils.generate_surrogate_key([...natural keys])`

## Singular vs plural

- Tables: **singular** (`fct_booking` is fine but we keep the brief's convention: `fct_bookings`)
- Measures: descriptive (`bookings_count`, `total_revenue`)
- Metrics: business-friendly singular (`load_factor`, `route_margin_pct`)

## File organisation

```
models/
├── staging/        stg_*.sql + _staging__models.yml
├── intermediate/   int_*.sql
│   └── nlp/        int_feedback_*.sql, int_route_complaint_themes.sql
├── marts/
│   ├── dim/        dim_*.sql
│   └── fct/        fct_*.sql
│   └── _marts__models.yml
├── semantic/       _semantic_models.yml + _metrics.yml + metricflow_time_spine.sql
└── ontology/       ont_*.sql + _ontology__models.yml
```

## Tests naming

- Generic tests (`unique`, `not_null`, `relationships`, `accepted_values`) → declared in `schema.yml`
- Singular tests in `tests/` → prefixed with `assert_` and describing the invariant

Examples:
- `assert_at_risk_customers_have_dissatisfaction_signal.sql`
- `assert_strategic_underperforming_routes_are_strategic.sql`
- `assert_loyal_detractor_is_gold.sql`
